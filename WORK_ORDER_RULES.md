# 工作單建立規則

## 工作單格式規範

每張工作單必須包含以下欄位：

### 1. 編號
- 格式：從 `0001` 開始，依序遞增
- 範例：`0001`, `0002`, `0003`...

### 2. 日期
- 格式：建立工作單的日期
- 範例：`2026-01-23`

### 3. 工作單標題
- 格式：簡潔明確的標題，描述該工作單的主要任務
- 範例：`建立專案基礎架構`、`實作牌組初始化功能`

### 4. 工單主旨
- 格式：說明要做什麼部分，通常是一個功能模組或開發階段
- 範例：`專案初始化`、`牌組管理模組`、`問牌功能實作`

### 5. 內容
- 格式：詳細描述工作內容，包括：
  - 具體要實作的功能
  - 需要建立的檔案
  - 需要實作的邏輯
  - 相關的技術要求
  - 驗收標準（可選）

## 工作單檔案命名規則

- 格式：`WORK_ORDER_XXXX.md`（XXXX 為編號）
- 範例：`WORK_ORDER_0001.md`, `WORK_ORDER_0002.md`

## 工作單存放位置

- 所有工作單存放在 `work_orders/` 目錄下

## 工作流程

### 單一工單處理流程

每張工單完成時，依序執行：

1. **執行工作單**：按照工作單內容進行開發
2. **單元測試**：執行相關單元測試，覆蓋率須達 **75% 以上**
3. **撰寫報告書**：測試通過後，撰寫完成報告
4. **本地 Commit**：將完成的工作提交到本地版本控制

### 計畫書層級處理流程

當一份計畫書下的**所有工單**都完成後，依序執行：

1. **同步更新計畫書**：更新計畫書狀態，標記已完成的項目
2. **整合測試**：執行該計畫書範圍內的整合測試
3. **本地 Commit**：將計畫書更新與整合測試結果提交到本地版本控制

### 計畫書狀態更新內容

更新計畫書時須包含：
- 各工單完成狀態（✓ 標記）
- 實際完成日期
- 重要變更或偏差記錄
- 整合測試結果摘要

## 報告書格式

每完成一份工作單後，需撰寫報告書，包含：
- 工作單編號
- 完成日期
- 完成內容摘要
- 遇到的問題與解決方案
- 測試結果
- 下一步計劃

## 報告書檔案命名規則

- 格式：`REPORT_XXXX.md`（XXXX 為工作單編號）
- 範例：`REPORT_0001.md`, `REPORT_0002.md`

## 報告書存放位置

- 所有報告書存放在 `reports/` 目錄下
