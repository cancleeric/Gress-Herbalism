## 一、現狀分析

### 程式碼審查結果

經過對前後端程式碼的完整審查，好友功能的基本流程已實作完成，但發現以下問題：

| 嚴重度 | 問題 | 位置 | 說明 |
|--------|------|------|------|
| P1 | 搜尋結果未過濾已加好友和已發送請求的玩家 | `friendService.js:13-28` | `searchPlayers` 只排除自己，未排除已是好友或已發送請求的人。使用者可能對已是好友的人重複發送請求 |
| P1 | 搜尋結果未過濾匿名玩家 | `friendService.js:13-28` | 匿名玩家（`firebase_uid` 為 NULL）也出現在搜尋結果中，但加好友無意義 |
| P2 | 後端測試缺少雙向互加自動接受場景 | `friendService.test.js` | `sendFriendRequest` 的 `autoAccepted` 分支無測試覆蓋 |
| P2 | 前端搜尋結果僅用前端移除已發送請求的玩家 | `Friends.js:88` | 重新搜尋後，已發送請求的玩家又會出現 |
| P2 | 好友列表缺少即時更新 | `Friends.js` | 接受好友請求後手動呼叫 `loadFriends`，若失敗則列表不更新 |

## 三、發現問題的修復計畫

### 工單 0178：後端 — 搜尋結果過濾優化

修復搜尋結果包含不應出現的玩家：

1. **排除已加好友**：在 `searchPlayers` 查詢中排除 friendships 表中已有關係的玩家
2. **排除已發送 pending 請求**：排除 friend_requests 表中 pending 狀態的對象
3. **排除匿名玩家**：只搜尋 `firebase_uid` 不為 NULL 的玩家
4. **新增後端測試**：覆蓋過濾邏輯

### 工單 0179：後端 — 新增雙向互加自動接受測試

補充缺失的測試覆蓋：

1. **新增 `sendFriendRequest` 雙向互加測試**：模擬 A 對 B 已有 pending 請求，B 再對 A 發送時自動接受
2. **驗證 autoAccepted 返回值**

### 工單 0180：功能驗證測試

執行所有既有測試，確認修改不影響現有功能。
