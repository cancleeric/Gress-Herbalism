# 桌遊網頁版專案規劃

## 專案概述
將一款3-4人玩的推理桌遊製作成網頁版，玩家需要透過問牌和推理來猜測桌面上的兩張蓋牌顏色。

## 遊戲規則摘要

> **完整規則請參考**：[遊戲規則完整說明](./docs/GAME_RULES.md)

### 基本規則
- **牌組構成**：總共14張牌（紅色2張、黃色3張、綠色4張、藍色5張）
- **遊戲設置**：洗牌後抽出2張蓋著放在桌面，剩餘牌平均發給所有玩家
- **勝利條件**：正確猜出桌上兩張蓋牌的顏色
- **遊戲流程**：輪流進行，每回合可選擇「問牌」或「猜牌」

### 問牌機制 ⚠️
- **動作組成**：選擇兩個顏色 → 選擇目標玩家 → 選擇要牌方式
- **要牌方式**（⚠️ **重要**：以下三種方式都是基於前面選定的兩個顏色來操作）：
  1. **兩個顏色各一張**：從選定的兩個顏色中**各要一張**（注意：是「各一張」，不是「各全部」）
     - 如果目標玩家兩個顏色都有 → 各給一張（共2張）
     - 如果只有一個顏色 → **只給該顏色一張**（共1張，**不是給該顏色的全部**，有多少給多少）
     - 如果都沒有 → 給「沒有」的回應
  2. **其中一種顏色全部**：從選定的兩個顏色中，選一個顏色要全部
     - 如果目標玩家有該顏色 → 給全部
     - 如果沒有 → 給「沒有」的回應
  3. **給一張要全部**：給目標玩家其中一個顏色一張，要另一個顏色的全部
     - 先給牌，然後要全部（如果沒有則返回「沒有」，但已給出的牌不收回）
- **重要規則**：
  - ⚠️ **玩家必須誠實給牌，不能拒絕**（除非真的沒有牌）
  - ⚠️ **有多少給多少**，不能選擇性給牌

### 猜牌機制 ⚠️
- **猜牌動作**：選擇兩個顏色作為猜測（顏色可以重複）
- **查看答案**：⚠️ **猜牌時，猜牌者可以查看正確答案**（蓋牌的實際顏色）
- **猜對處理**：
  - 該玩家獲勝，遊戲結束
  - ⚠️ **公布正確答案給所有玩家**（顯示蓋牌的實際顏色）
- **猜錯處理**：
  - 該玩家退出遊戲
  - ⚠️ **將兩張蓋牌蓋回桌面**（保持隱藏狀態）
  - ⚠️ **如果只剩一個玩家且猜錯**：遊戲結束，**沒有獲勝者**
- **特殊情況**：⚠️ **如果只剩一個玩家，該玩家必須執行猜牌動作**（不能選擇問牌）

## 核心功能模組

### 1. 遊戲房間管理
- 創建房間
- 加入房間（3-4人限制）
- 離開房間
- 房間狀態同步

### 2. 牌組管理
- 牌組初始化（14張牌：紅2、黃3、綠4、藍5）
- 洗牌功能
- 發牌邏輯（抽出2張蓋牌，剩餘平均分配）
- 手牌管理（每個玩家只能看到自己的牌）

### 3. 遊戲流程控制
- 玩家輪流機制
- 回合狀態管理（問牌/猜牌階段）
- 遊戲結束判定

### 4. 問牌功能
- 顏色選擇（從紅、黃、綠、藍中選兩個不同顏色）⚠️ **先選顏色**
- 目標玩家選擇
- 要牌方式選擇（⚠️ **基於前面選定的兩個顏色**）：
  - 方式一：兩個顏色各一張（從選定的兩個顏色中**各要一張**，注意：是「各一張」，不是「各全部」）
    - 如果目標玩家兩個顏色都有 → 各給一張（共2張）
    - 如果只有一個顏色 → **只給該顏色一張**（共1張，**不是給該顏色的全部**，有多少給多少）
    - 如果都沒有 → 給「沒有」的回應
  - 方式二：其中一種顏色全部（從選定的兩個顏色中，選一個顏色要全部）
    - 如果目標玩家有該顏色 → 給全部
    - 如果沒有 → 給「沒有」的回應
  - 方式三：給其中一種顏色一張，要另一種顏色全部
    - 先給牌，然後要全部（如果沒有則返回「沒有」，但已給出的牌不收回）
- ⚠️ **問牌重要規則**：
  - 玩家必須誠實給牌，不能拒絕（除非真的沒有牌）
  - 有多少給多少，不能選擇性給牌
  - 問牌結果處理（包括「沒有」的回應）

### 5. 猜牌功能
- 猜牌介面（選擇兩個顏色，可以重複）
- ⚠️ **猜牌時，猜牌者可以查看正確答案**（蓋牌的實際顏色）
- 猜牌結果驗證
- **猜對處理**：
  - 遊戲結束，顯示獲勝者
  - ⚠️ **公布正確答案給所有玩家**（顯示蓋牌的實際顏色）
- **猜錯處理**：
  - 該玩家退出遊戲
  - ⚠️ **將兩張蓋牌蓋回桌面**（保持隱藏狀態）
  - 如果還有其他玩家：繼續遊戲
  - ⚠️ **如果只剩一個玩家且猜錯**：遊戲結束，**沒有獲勝者**
- ⚠️ **特殊情況**：如果只剩一個玩家，該玩家**必須執行猜牌動作**（不能選擇問牌）

### 6. 遊戲狀態同步
- 實時更新所有玩家的遊戲狀態
- 手牌變化同步
- 桌面蓋牌狀態（不顯示內容，只顯示位置）

## 技術架構建議

### 方案一：純前端（適合本地多人遊戲）
- **框架**：React 或 Vue.js
- **狀態管理**：Redux 或 Vuex
- **樣式**：CSS Modules 或 Tailwind CSS
- **優點**：簡單快速，無需後端
- **缺點**：無法遠程多人遊戲

### 方案二：前後端分離（適合線上多人遊戲）
- **前端**：React 或 Vue.js
- **後端**：Node.js + Express 或 Python + Flask
- **即時通訊**：WebSocket (Socket.io)
- **數據庫**：MongoDB 或 PostgreSQL（存儲遊戲記錄）
- **優點**：支援遠程多人遊戲，可擴展
- **缺點**：開發複雜度較高

### 方案三：混合方案（推薦）
- **前端**：React 或 Vue.js
- **即時通訊**：WebSocket (Socket.io) 或 Firebase Realtime Database
- **狀態管理**：前端狀態管理 + 後端驗證
- **優點**：平衡開發複雜度和功能需求

## 數據結構設計

### 遊戲狀態 (GameState)
```javascript
{
  gameId: string,
  players: [
    {
      id: string,
      name: string,
      hand: Card[],      // 手牌（只有自己可見）
      isActive: boolean, // 是否仍在遊戲中
      isCurrentTurn: boolean
    }
  ],
  hiddenCards: Card[],   // 桌上兩張蓋牌（不顯示顏色）
  currentPlayerIndex: number,
  gamePhase: 'waiting' | 'playing' | 'finished',
  winner: string | null,
  gameHistory: Action[]  // 遊戲歷史記錄
}
```

### 牌卡 (Card)
```javascript
{
  id: string,
  color: 'red' | 'yellow' | 'green' | 'blue',
  isHidden: boolean  // 是否為蓋牌
}
```

### 問牌動作 (QuestionAction)
```javascript
{
  type: 'question',
  playerId: string,
  targetPlayerId: string,
  colors: ['red' | 'yellow' | 'green' | 'blue', 'red' | 'yellow' | 'green' | 'blue'],
  questionType: 1 | 2 | 3,  // 三種要牌方式
  result: {
    cardsGiven: Card[],
    cardsReceived: Card[]
  }
}
```

### 猜牌動作 (GuessAction)
```javascript
{
  type: 'guess',
  playerId: string,
  guessedColors: ['red' | 'yellow' | 'green' | 'blue', 'red' | 'yellow' | 'green' | 'blue'],
  isCorrect: boolean
}
```

## UI/UX 設計要點

1. **遊戲大廳**
   - 顯示房間列表或創建新房間
   - 顯示房間內玩家列表
   - 開始遊戲按鈕（當人數達到3-4人時）

2. **遊戲介面**
   - 中央：顯示兩張蓋牌位置（背面圖示）
   - 下方：玩家自己的手牌（顯示顏色）
   - 側邊：其他玩家資訊（不顯示手牌）
   - 操作區：問牌/猜牌按鈕和選項

3. **問牌介面**
   - 顏色選擇器（兩個不同顏色）
   - 玩家選擇器
   - 要牌方式選擇（三種方式）
   - 確認/取消按鈕

4. **猜牌介面**
   - 顏色選擇器（兩個顏色，可重複）
   - 確認猜測按鈕
   - 警告提示（猜錯會退出）

5. **遊戲狀態顯示**
   - 當前輪到誰
   - 遊戲歷史記錄
   - 玩家狀態（活躍/已退出）

## 代碼可擴展性設計 ⚠️ **重要**

為了方便未來擴充功能，專案將採用以下設計原則：

### 1. 模組化架構
- **功能分離**：每個功能模組獨立，互不依賴
- **組件化設計**：UI 組件可重用、可替換
- **服務層分離**：業務邏輯與 UI 分離，便於修改和測試

### 2. 配置化設計
- **遊戲規則配置化**：牌組配置、規則參數可通過配置文件修改
- **常數集中管理**：所有遊戲常數（顏色、牌數等）集中在 `constants.js`
- **擴展點設計**：預留擴展接口，新增功能時只需實現接口

### 3. 數據結構可擴展
- **使用 TypeScript 或 JSDoc**：明確數據結構，便於擴展
- **版本化數據結構**：遊戲狀態支持版本號，便於未來升級
- **向後兼容**：新功能不破壞現有功能

### 4. 插件化架構（可選）
- **動作系統**：問牌、猜牌等動作作為可擴展的動作類型
- **規則引擎**：遊戲規則可通過規則引擎配置，無需修改核心代碼
- **事件系統**：使用事件驅動，方便添加新的事件監聽和處理

### 5. 代碼組織建議
```
utils/
├── cardUtils.js          # 牌組工具（可擴展支援新牌型）
├── gameRules.js          # 遊戲規則（規則驗證邏輯，易於修改）
└── actionHandlers/       # 動作處理器（每個動作類型一個文件）
    ├── questionAction.js
    ├── guessAction.js
    └── [未來新動作].js   # 新增動作只需添加新文件

services/
├── gameService.js        # 核心遊戲服務（保持穩定）
└── extensions/           # 擴展功能服務
    └── [未來新功能].js
```

### 6. 擴展性標記
- 所有可能擴展的地方會加上 `// TODO: 可擴展點` 註釋
- 使用策略模式處理不同的要牌方式，方便新增方式
- 使用工廠模式創建動作，方便新增動作類型
